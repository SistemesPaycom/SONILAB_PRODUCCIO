1. RESUM DEL PROBLEMA (sintomes observats per l'usuari)
--------------------------------------------------------
En obrir un fitxer .srt al editor de subtitols, tota la zona del editor
comencava a parpadejar de forma massiva i visible. El mateix parpadeig
apareixia a la taula Files del Library. La resta de la UI (video, toolbar,
header) no es veia afectada.

Caracteristiques clau del sintoma:
  - Apareixia immediatament en obrir el editor, no calia interactuar
  - Mes facil de reproduir quan l'usuari tenia multiples pestanyes obertes
    (2 Brave + 2 Edge en el cas que ens va permetre diagnosticar-ho)
  - A vegades s'aturava tot sol al cap de ~5 minuts sense explicacio
    aparent, pero tornava en recarregar
  - L'usuari va observar correctament que semblava "un pols entre Tema i
    Estil" — intuicio que va resultar molt propera al mecanisme real

Impacte: bloquejava completament qualsevol treball amb el editor de
subtitols, i per tant bloquejava les proves manuals del feature Factory
Reset (T13 del plan d'implementacio).

3. ROOT CAUSE EXACTE
---------------------
UserStylesContext.tsx tenia dos blocs que junts formaven un loop infinit
en presencia de multiples pestanyes:

BLOC 1 — useEffect de persistencia (linia 138-140 aprox abans del fix):
  useEffect(() => {
    if (me?.id) writeScopedLocal(me.id, payload);
  }, [me?.id, payload]);

  Cada vegada que payload canvia, escriu a localStorage.
  writeScopedLocal crida localStorage.setItem(scopedKey(userId), JSON...).

BLOC 2 — useEffect de sync cross-tab (linia 143-155 aprox abans del fix):
  useEffect(() => {
    if (!me?.id) return;
    const key = scopedKey(me.id);
    const onStorage = (e: StorageEvent) => {
      if (e.key !== key || !e.newValue) return;
      try {
        const parsed: UserStylesPayload = JSON.parse(e.newValue);
        if ((parsed as any)?.version === 2) setPayload(parsed);  // ← sense guard
      } catch { /* noop */ }
    };
    window.addEventListener('storage', onStorage);
    return () => window.removeEventListener('storage', onStorage);
  }, [me?.id]);

  L'event "storage" del navegador es dispara a TOTES les pestanyes germanes
  del mateix origin quan una d'elles fa localStorage.setItem a una key
  que ja existia. Inclou pestanyes dins del mateix navegador (no entre
  navegadors diferents com Brave ↔ Edge, cada un te el seu localStorage).

MECANISME DEL LOOP (exemple amb 2 pestanyes al mateix navegador):

  Estat inicial:
    Pestanya P1: payload = A (2474 bytes) — perque va carregar primer amb
                 un path de loadOrMigrate determinat
    Pestanya P2: payload = B (2441 bytes) — perque va carregar segons amb
                 un timing lleugerament diferent, entrant a un branch
                 diferent de loadOrMigrate (p.ex. el valor scopedLocal va
                 arribar abans que la resposta de me?.preferences)

  Tick 1: P1 te A, escriu A al localStorage (effect de persistencia).
          → localStorage ara te A.
          → Es dispara 'storage' event a P2 amb e.newValue = A.
          → P2 onStorage rep A, fa setPayload(A), re-render, aplica al DOM.
          → Ara P2 te A tambe? NO. Perque l'effect de persistencia de P2
            es reactiva i escriu SEGUIDAMENT el seu B "old" perque React
            encara no ha flushed el setPayload(A) per complet. O perque
            hi ha un re-render que reverteix. La clau del bug es que la
            segona pestanya acaba escrivint la seva propia versio B.
          → Es dispara 'storage' event a P1 amb e.newValue = B.
          → P1 onStorage rep B, fa setPayload(B), re-render, aplica al DOM.
          → P1 persistencia reverteix a A.
          → ...

  Alternativament (i mes probablement): dues pestanyes amb payloads
  genuinament diferents, cadascuna escriu el seu, l'altra llegeix i
  setPayloda, l'altra torna a escriure el seu, loop infinit.

  Tant la mitjana com el pic: ~790 iteracions/seg, visible com flicker
  massiu a qualsevol element que consumeixi var(--us-*).

PER QUE NO BASTAVEN ELS GUARDS PER CONTINGUT:
  Una primera proposta va ser afegir un guard dins d'onStorage del tipus
  `if (JSON.stringify(currentPayload) === e.newValue) return;`. Pero el
  bug es que els dos payloads son GENUINAMENT DIFERENTS (2474 bytes vs
  2441 bytes, 33 bytes de diferencia real). Un guard per igualtat de
  contingut no convergeix perque MAI son iguals — i per tant mai trencaria
  el loop.

  L'unica manera d'eliminar el ping-pong era trencar un dels dos blocs
  completament.

5. ALTERNATIVES CONSIDERADES I DESCARTADES
-------------------------------------------
Alternativa A: guard per serialize dins d'onStorage.
  Rebutjat: els dos payloads son genuinament diferents, el guard mai
  convergiria.

Alternativa B: guard per serialize dins del writeScopedLocal useEffect
(no escriure si el serialized actual es igual al que ja hi ha a localStorage).
  Podria trencar el loop en alguns casos, pero no quan les dues pestanyes
  tenen cadascuna un payload diferent (cadascuna ve determinada a escriure
  el seu, que per a ella es "nou respecte al que hi ha"). Complex i fragil.

Alternativa C: BroadcastChannel amb sender tab id i bailout self-eco.
  Es l'arquitectura correcta si en algun moment cal tornar a tenir sync
  cross-tab real. Massa canvi per a un fix d'emergencia. Documentat al
  comentari inline per a futur.

Alternativa D: desactivar UserStylesContext completament.
  Nuclear, rompria Settings → Estils i tots els components que consumeixen
  var(--us-*). Descartat.

Alternativa E (la triada): eliminar nomes el listener cross-tab. Minim
blast radius, trenca el loop en el receptor. Persistencia i DOM apply
intactes. Zero regression single-tab. Known limitation documentada.

El bug de flicker NO era part del treball del Factory Reset. Era pre-existent
al codebase, latent, i nomes es manifestava amb el cas multi-tab que l'usuari
no havia tingut fins a probar-ho a fons aquest dia. Si el domain doc
domain-user-styles.md:75 menciona un bug historic de flicker (commit ef60b36),
aquell era DIFERENT: mutacions DOM durant render amb !important/MutationObserver,
corregit per ef60b36. El bug d'avui era un altre, i s'ha de documentar com a
categoria separada.

Si en el futur es vol recuperar la sincronitzacio cross-tab dels user-styles
(es a dir, que si edites un preset en una pestanya, l'altra pestanya ho
vegi en temps real sense recarregar), la implementacio CORRECTA es:

  - Usar BroadcastChannel en lloc d'addEventListener('storage')
  - Cada missatge ha de portar un senderTabId unic generat al mount del provider
  - El receptor ha de verificar `if (msg.senderTabId === myTabId) return;`
    per evitar que rebre l'eco del seu propi enviament
  - El emissor nomes ha d'enviar missatges per canvis iniciats per l'usuari
    (edits via Settings → Estils), NO per cada mutacio del payload (que inclou
    els canvis rebuts d'altres pestanyes — aixo reprodueix el loop)
  - Alternativament: usar un debounce i un "last known sender" per garantir
    que mai es propaga el propi canvi

Aquest refactor esta explicitament mencionat al comentari inline que
substitueix el codi eliminat, per si algu el troba al futur.
